# 远程授权上线前复盘

本文档用于总结本次 `remote_core / verified_loader` 远程授权链路在本地联调中暴露出的主要错误、根因和正式版上线前必须规避的事项。后续正式上线、灰度验证、批量迁移真实脚本时，统一先对照本文执行。

## 1. 结论先行

本次联调结果说明：

1. 授权审批、授权码、单机绑定、远程模块下发的大方向可行。
2. 真正困难的不是数据库和接口本身，而是“浏览器脚本运行环境 + 远程授权 UI + HTTPS/跨域链路 + 多标签页行为”。
3. 正式版上线前，必须把“脚本能跑”提升到“用户知道发生了什么、知道下一步该做什么、后台能准确看到过程”。

## 2. 本次踩过的主要错误

### 2.1 把运行时地址直接写成本地 `http://127.0.0.1:xxxx`

#### 现象

- 脚本页面能弹出授权框。
- 点击“提交申请”看起来没有任何反应。
- 后台收不到申请。

#### 根因

- 目标网页是 `https://mooc.ctt.cn/...`
- 脚本运行时请求的却是本地 `http://127.0.0.1:xxxx`
- 浏览器把 HTTPS 页面访问 HTTP 资源视为混合内容，直接拦截。

#### 有效做法

- 正式版绝不能把运行时地址写成本地 HTTP。
- 正式环境必须使用 HTTPS 域名。
- 本地调试时，如目标站点是 HTTPS，必须使用 HTTPS 隧道或正式 HTTPS 域名。

#### 上线要求

- 后台运行时主地址只允许填 HTTPS 地址。
- 下载脚本前若检测到运行时地址不是 HTTPS，应提示管理员风险，不建议发布。

## 3. 脚本头部与安装识别错误

### 3.1 远程核心壳脚本仍保留旧脚本名，导致安装识别混乱

#### 现象

- 下载的是远程核心壳脚本，但 Tampermonkey 里名字看起来像旧脚本。
- 用户难以判断自己装的是老版本还是新壳脚本。

#### 根因

- 生成壳脚本时直接沿用了原始 UserScript 头部。
- `@name`、`@description` 没有明确区分远程核心模式。

#### 有效做法

- `remote_core` 和 `verified_loader` 下载时必须重写头部元信息。
- 脚本名、描述、版本、授权能力、运行时标识都要明确可见。

#### 上线要求

- `@name` 必须可区分模式。
- 正式版命名不要出现 `Remote Core Remote Core` 这类临时调试命名错误。

## 4. 授权 UI 交互设计错误

### 4.1 早期只用 `prompt/alert`，用户几乎无法理解流程

#### 现象

- 用户不知道什么时候该输入授权码。
- 用户不知道没授权时应该如何申请。
- 脚本报错时几乎没有可见提示。

#### 根因

- 浏览器脚本早期用了最低成本的 `prompt/alert`。
- 没有页面内可见状态面板。

#### 有效做法

- 必须使用页面内固定授权面板。
- 至少显示：
  - 当前状态
  - 输入授权码
  - 提交申请
  - 成功/失败提示
  - 当前正在执行的阶段

### 4.2 “提交申请”和“保存并继续”流程早期互相打架

#### 现象

- 点“提交申请”只切换了表单，主流程却还卡在等待授权码。
- 点“保存并继续”也没有明确状态变化。

#### 根因

- 授权码输入 Promise 和申请表单 Promise 的切换逻辑设计有误。
- UI 切过去了，但主流程没有真正继续执行到正确分支。

#### 有效做法

- “输入授权码”和“提交申请”必须是明确的两条动作链。
- 点“提交申请”要立即退出等待授权码阶段并进入申请提交逻辑。
- 点“保存并继续”要立即进入激活和校验逻辑，并有状态提示。

### 4.3 后台审批通过后，脚本前端没有同步感知

#### 现象

- 用户在前端提交申请后，后台已审批并生成授权码。
- 但前端面板仍提示“等待审批”，用户以为没有通过。
- 实际输入授权码后可以正常使用。

#### 根因

- 早期授权面板只负责“提交申请”和“输入授权码”，没有轮询后台审批状态。
- 面板状态和后台真实审批状态不同步。

#### 有效做法

- 申请提交后，前端必须轮询申请状态。
- 一旦后台审批通过，应自动把面板切回授权码输入态，并提示用户“审批已通过，请输入授权码继续”。
- 更优做法是审批通过后直接自动写入授权信息并自动继续激活，不再要求用户手抄授权码。

## 5. 生成器转义错误

### 5.1 壳脚本生成后 JS 被拼坏

#### 现象

- 用户装上脚本后页面完全没反应。
- 代码中能看到异常片段，例如：
  - `replace(//$/...)`
  - `replace(/^s*exports+/...)`
  - `new Function(...` 被断行拼坏

#### 根因

- 字符串模板里嵌正则和换行时，转义层级没处理好。
- 生成器在 Node 中看起来正确，但输出到最终脚本后变成非法 JS。

#### 有效做法

- 所有壳脚本生成逻辑必须经过“最终产物文本检查”。
- 不能只检查生成器源码本身，要检查下载出来的最终脚本。

#### 上线要求

- 每次发布前必须抽样检查下载脚本文本中是否包含明显坏片段。

## 6. 授权通过后多标签页行为暴露的问题

### 6.1 每打开一个课程新标签页，授权面板都会再次弹出，体验突兀

#### 现象

- 用户已授权成功。
- 但每个新标签页进入课程时仍看到授权与运行时面板。

#### 根因

- 当前脚本是每个标签页单独初始化。
- 运行时面板默认同步显示，没有对“已授权且已绑定”的场景做更平滑的静默通过。

#### 有效做法

- 已授权用户再次打开新页时，应优先静默校验。
- 只有在以下场景才主动弹面板：
  - 没有授权码
  - 授权码失效
  - 设备不匹配
  - 运行时连接失败
  - 远程模块拉取失败
- 对“授权成功、正在拉取模块”这种短状态，优先做轻提示而不是大面板常驻。

#### 正式版建议

- 授权面板分为：
  - 阻断式面板：仅未授权/报错时出现
  - 非阻断轻提示：授权成功、拉取中、同步中时出现

## 7. 后台管理页过于堆叠

### 7.1 一个页面同时塞运行时、远程核心、授权、设备、会话、事件，理解成本高

#### 现象

- 管理员进入“授权与运行时”页后觉得内容多、布局乱、不知道先看哪里。

#### 根因

- 当前阶段为了开发联调方便，把多个子能力堆在一个页里。
- 缺少正式产品化的信息分层。

#### 有效做法

- 正式版应至少拆分为：
  1. 运行时配置
  2. 远程核心管理
  3. 授权审批
  4. 授权码与设备
  5. 会话与事件

## 8. 正式版上线前必须完成的改进

### 8.1 脚本侧

1. 远程核心脚本头部命名规范化。
2. 授权面板区分“首次申请”和“已授权静默校验”。
3. 授权成功后默认收起或降级为轻提示，不要常驻大面板。
4. 每个标签页只在必要时打断用户。

### 8.2 后台侧

1. 授权与运行时页面拆分或重构布局。
2. 审批通过后可一键复制授权码并给出用户操作提示。
3. 已授权设备、活跃会话、最近错误要更醒目。

### 8.3 发布与验证侧

1. 正式环境必须使用 HTTPS 运行时域名。
2. 发布前抽样下载脚本，检查最终产物文本是否正常。
3. 真实浏览器 + Tampermonkey 环境中完整验证：
   - 提交申请
   - 后台审批
   - 输入授权码
   - 首次激活绑定
   - 新标签页打开课程
   - 设备绑定/会话上报
   - 错误提示

## 9. 本次联调沉淀的最终经验

1. 远程授权系统最容易出问题的不是“后端接口”，而是“浏览器脚本真实运行环境”。
2. 只要目标站点是 HTTPS，本地调试就不能再用 HTTP 运行时地址。
3. 油猴脚本必须以“最终下载产物”为准做检查，不能只看生成器源码。
4. 对用户来说，“知道现在发生了什么”比“后台功能很多”更重要。
5. 正式版一定要把授权体验做成：
   - 未授权时明确阻断
   - 已授权时尽量无感
   - 失败时明确可见
