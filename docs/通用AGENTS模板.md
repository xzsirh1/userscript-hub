# 通用 AGENTS.md 模板

> 使用方式：新项目创建后，把本文件内容复制到项目根目录 `AGENTS.md`，再把所有 `<...>` 占位符替换成真实项目情况。  
> 原则：这是一份“给 AI 编码代理看的项目宪法”，目标是让后续 AI 少乱改、少丢数据、少重复踩坑。

# 项目级 AI 规则

- 作用范围：整个项目。
- 优先级：高于一般习惯、旧聊天记录和零散说明。
- 维护原则：如果出现新的规则、新经验、新踩坑结论，优先更新本文件。
- 冲突处理：如果本文件和用户当前明确指令冲突，以用户当前明确指令为准，但不得违反数据安全、隐私和破坏性操作底线。

## 2. 总体原则

### 2.1 最小修改原则

- 只修改完成当前需求所必需的文件。
- 不顺手重构无关代码。
- 不新增第二套平行实现，能复用现有逻辑就复用。
- 不因为“看起来更优雅”而改变已有稳定行为。

### 2.2 数据安全第一

- 永远不要删除、覆盖、移动真实用户数据，除非用户明确要求且已说明风险。
- 默认禁止破坏这些目录或文件：
  - `data/`
  - `uploads/`
  - `server/data/`
  - `server/uploads/`
  - `.env`
  - `.git/`
  - 数据库文件，如 `*.db`、`*.sqlite`
  - 用户上传、导入、备份、生成的业务文件
- 派生数据可以重建，原始数据不可随意破坏。
- 清理临时文件前必须确认路径在预期临时目录内。

### 2.3 向后兼容原则

- 新功能不能破坏已有功能、旧数据、旧配置和旧接口。
- API 修改优先保持兼容；确需破坏兼容时，必须说明迁移方式。
- 数据库结构变更必须提供迁移逻辑、默认值和旧库兼容处理。
- 配置项变更必须提供默认值，不能让老用户升级后直接崩溃。

### 2.4 安全优先原则

- 所有外部输入都要验证。
- 数据库查询必须参数化，避免 SQL 注入。
- 文件上传必须校验类型、大小、路径和权限。
- 文件下载必须校验记录、路径归属和访问权限。
- 敏感信息不能硬编码，不能写入文档、日志、更新包或提交记录。
- 错误信息不要向用户暴露密钥、服务器真实路径、内部堆栈和隐私数据。

### 2.5 可诊断原则

- 不要把所有失败都说成“请求失败”或“AI 调用失败”。
- 日志应尽量说明：失败阶段、错误类型、上游状态码、关键参数规模、是否可重试、是否已有部分结果。
- 日志不能记录 API Key、密码、Cookie、Token、完整隐私内容或模型思维链。
- 如果无法确定根因，优先增加诊断日志和复现步骤，不要盲猜。

### 2.6 文档同步原则

代码行为变化后，必须按需同步更新：

- `README.md`
- `CHANGELOG.md`
- `docs/`
- `AGENTS.md`

如果某个经验会帮助未来 AI 少踩坑，就不要只留在聊天记录里，必须写进文档。

## 3. Git 与分支规则

- 修改前先查看工作区状态：`git status --short`。
- 不要回滚用户未要求回滚的改动。
- 不要使用 `git reset --hard`、`git checkout -- .` 等破坏性命令，除非用户明确要求。
- 不要 amend 已有提交，除非用户明确要求。
- 提交信息要说明实际变化，例如：
  - `Fix setup initialization flow`
  - `Improve remote core download validation`
  - `Add deployment checklist`
- 发布到 GitHub 前必须检查：
  - `.env` 未提交
  - 数据库未提交
  - 上传文件未提交
  - 构建临时目录未提交
  - 真实服务器 IP、密码、Token 未出现在公开文件中

## 4. 目录归属规则

根目录只放项目入口、配置和必要说明。不要随手把临时文件、测试文件、压缩包堆在根目录。

推荐归属：

- 新文档放 `docs/`
- 工具脚本放 `scripts/`
- 测试或审计临时文件放 `<临时目录>`，例如 `build/temp/` 或 `artifacts/build-staging/`
- 正式发布包放 `<发布包目录>`，例如 `build/archives/` 或 `artifacts/releases/`
- 用户数据放 `data/` 或 `server/data/`
- 上传文件放 `uploads/` 或 `server/uploads/`

如果新增顶层目录，必须同步更新 `README.md` 和 `AGENTS.md`。

## 5. 版本号规则

如果项目有版本号，每次用户可见变化都必须更新：

- `package.json`
- `version.json`
- `CHANGELOG.md`

版本号建议遵循语义化版本：

- `MAJOR`：不兼容变化。
- `MINOR`：新增功能，向后兼容。
- `PATCH`：Bug 修复、文档、兼容性小修。

如果修改 `package.json` 依赖或版本，按项目技术栈更新锁文件，例如：

```bash
npm install --package-lock-only
```

## 6. 开发流程

### 6.1 需求分析

动手前先判断：

- 需求影响哪些页面、接口、数据库、配置、部署方式？
- 是否涉及用户数据？
- 是否涉及权限、安全、文件上传下载？
- 是否涉及更新包、安装包、发布包？
- 是否需要兼容旧版本？

### 6.2 编码顺序

通用建议：

1. 数据结构或模型层。
2. 后端服务 / 控制器。
3. API 路由。
4. 前端 API 封装。
5. 前端页面与交互。
6. 路由、菜单、入口。
7. 文档和变更记录。

如果项目有自己的顺序，以本文件项目专属章节为准。

### 6.3 修改要求

- 优先使用现有风格和现有组件。
- 保持错误提示清晰，用户能知道下一步怎么做。
- 前端至少考虑桌面和手机宽度。
- 后端要处理成功、失败、空数据、权限不足、资源不存在等情况。
- 涉及长任务时，不要阻塞普通页面操作；要有进度、日志、可重试机制。

## 7. 测试与验证规则

根据改动选择执行：

- 后端语法检查。
- 前端构建。
- 单元测试或集成测试。
- 关键接口验证。
- 关键页面浏览器验证。
- 数据库迁移验证。
- 文件上传 / 下载验证。
- 权限与错误场景验证。
- 移动端布局验证。

交付时必须说明：

- 运行了哪些检查。
- 哪些检查通过。
- 哪些检查没有运行，以及原因。
- 是否有残留风险。

## 9. 部署规则

部署前：

- 备份数据库和上传文件。
- 确认目标路径。
- 确认服务进程名或容器名，不要凭记忆猜。
- 确认端口、反向代理、上传大小限制、超时时间。

部署后：

- 检查服务是否启动。
- 检查健康接口。
- 检查首页和后台。
- 检查关键业务流程。
- 检查日志没有持续报错。

如果部署到正式服务器，记录：

- 部署包路径。
- 部署时间。
- 服务状态。
- 验证结果。
- 回滚方式。

不要把真实服务器账号、密码、Token 写进仓库。

## 10. 数据库与迁移规则

- 新增表、字段、索引必须写迁移逻辑。
- 迁移必须可重复执行或能安全跳过已存在结构。
- 大量数据迁移前要考虑备份和失败恢复。
- 删除字段或表必须特别谨慎，优先软迁移。
- 查询要避免 N+1。
- 大列表必须分页或限制数量。

## 11. 文件上传与下载规则

上传：

- 校验文件大小。
- 校验扩展名和 MIME。
- 校验保存路径，防止路径穿越。
- 不信任前端传来的文件名。

下载：

- 优先根据数据库记录定位文件。
- 校验文件存在且在允许目录内。
- 私有文件必须校验权限。
- 回源、同步或缓存逻辑要记录日志，方便排查。

## 12. 前端规则

- 保持现有设计系统和组件风格。
- 页面必须处理 loading、empty、error、success 状态。
- 表单必须有前端体验校验，后端仍要兜底校验。
- 401 引导重新登录。
- 403 提示无权限。
- 404 提示资源不存在。
- 500 提示失败并引导重试或查看日志。
- 手机端不能横向溢出，关键操作不能被侧边栏或浮层挡住。

## 13. 后端规则

- 路由分清公开接口和认证接口。
- 鉴权中间件不能被绕过。
- 控制器统一返回格式。
- 关键操作要记录日志。
- 生产环境错误信息要降噪，开发环境可保留更多诊断。
- 长任务要有状态、进度、日志和重试能力。

## 14. AI / LLM 功能规则

如果项目涉及 AI、LLM、向量、记忆、总结、结构化输出：

- 原始用户数据是最高事实源。
- AI 输出是派生数据，必须可删除、可重建、可追溯。
- 不要把低质量、证据不足、截断或解析失败的 AI 输出标记为成功。
- JSON 任务要有解析、修复、质量门禁和失败日志。
- 重试要带精简失败反馈，不要盲目重复同一个请求。
- 不记录 API Key、完整隐私内容或模型思维链。
- 模型失败时要区分网络、认证、限流、超时、HTML 代理页、JSON 解析、质量门禁等原因。
- 长文本任务必须分阶段、可恢复；某阶段失败时不能把半成品冒充完整成功。

## 16. 成功经验库

> 每次出现“已验证成功、已修复成功、已确认有效”的经验，都应补充到这里。  
> 记录格式建议：场景、根因、有效做法、验证方式。

### 16.1 示例：更新包规则收敛

- 场景：多人或多个 AI 维护时，容易出现多个打包脚本和多种压缩方式。
- 根因：没有唯一标准入口。
- 有效做法：只保留一个标准打包脚本，正式包和临时目录固定归属。
- 验证方式：检查根目录不再出现散落临时包，交付包结构稳定。

### 16.2 示例：数据不可破坏

- 场景：调试、打包、部署时误删用户数据。
- 根因：没有区分源码、构建产物、运行期数据。
- 有效做法：`data/`、`uploads/`、数据库和 `.env` 默认禁止删除、禁止打包、禁止提交。
- 验证方式：打包后检查压缩包，Git 提交前检查候选文件。
