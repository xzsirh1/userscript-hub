# 更新包说明

本文件只保留当前有效的更新包标准，不再记录某一次版本的临时说明。

## 标准流程

### 生成更新包

必须遵循以下顺序：

1. 先本地构建前端
2. 使用 `node scripts/create-update.js` 打包
3. 检查 ZIP 结构是否完整
4. 再上传到后台执行在线更新

当前规则补充：

- `scripts/create-update.js` 是唯一标准打包脚本
- 后续 AI 或人工若需调整打包逻辑，只能修改这一个文件
- 不再新增 `bat`、`sh`、`ps1` 等并行打包脚本，避免出现多套规则

### 禁止做法

- 不要手工随意压缩目录交付
- 不要交付缺少 `client/dist/` 的包
- 修改更新逻辑后，不要跳过构建和结构检查

## ZIP 标准结构

更新包内至少要有：

```text
server/src/...
server/package.json
server/ecosystem.config.js
client/src/...
client/package.json
client/dist/...
scripts/...
docs/...
```

## Docker 与 PM2 的差异

### Docker

- 更新目标：
  - `server/src/* -> /app/src/*`
  - `server/package.json -> /app/package.json`
  - `client/dist/* -> /app/public/*`
- Docker 默认不依赖容器内 `client` 源码重新构建
- 更新完成后由容器重启策略拉起，不依赖容器内执行 `docker restart`

### PM2 / 非 Docker

- 更新目标：
  - `server/src/* -> server/src/*`
  - `client/dist/* -> client/dist/*`

## 更新前检查

- 当前站点可正常访问
- 已导出一份备份包
- 磁盘空间足够
- Docker 与源站如涉及同步回源，准备同时升级到同版本

## 更新后验证

至少验证以下内容：

1. 首页和后台能正常打开
2. 脚本、插件、小程序下载正常
3. 网站设置中的导出功能正常
4. 数据同步可正常执行
5. Docker 环境下检查容器是否已自动重启并加载新代码

## 回滚建议

- 后台优先使用代码备份回滚
- 如果在线更新器本身异常，再使用宿主机脚本或手工覆盖回滚
- Docker 回滚后仍要检查 `/app/public` 和 `/app/src` 是否同步回退

## 常见问题

### Docker 更新后页面空白或接口不存在

- 优先排查前后端版本错配
- 检查 `client/dist` 是否真的进入了容器的 `/app/public`

### Docker 下载提示文件不存在

- 先确认本地 `uploads` 是否有文件
- 再确认源站与拉取站是否升级到同一版本
- 必要时执行一次全量同步补齐历史文件

### 导出完成但浏览器不立即下载

- 这类问题通常应改成浏览器直接下载，不要先在前端整包 `fetch + blob`
