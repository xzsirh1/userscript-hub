# 远程核心模块接入规范

本文档用于约束后续脚本作者在 `remote_core` 模式下的接入方式。

## 1. 模式说明

- `remote_core` 模式下，前台安装的是壳脚本，不直接下发完整业务代码。
- 壳脚本先连接运行时服务，获取 Manifest、远程配置和远程模块，再执行模块入口。
- 脚本作者后续开发新脚本时，必须按本文档的入口规范拆分代码。

## 2. 必须遵守的入口约定

远程模块至少导出以下函数：

```js
export async function bootstrap(context) {}
```

可选导出：

```js
export async function destroy(context) {}
export function getHealth() {}
```

要求：

- `bootstrap(context)` 必须存在
- `bootstrap(context)` 必须可重复执行，不依赖一次性全局污染
- 不得在模块内部硬编码服务端地址
- 所有远程配置统一从 `context.runtimeConfig` 读取

## 3. context 说明

运行时目前提供的 `context` 包括：

- `context.script`
  - `id`
  - `name`
  - `version`
- `context.authorizationCode`
- `context.runtimeConfig`
  - `manifestVersion`
  - `activeModuleVersion`
  - 以及后台维护的扩展配置
- `context.logger`
  - `info(message, payload?)`
  - `error(message, payload?)`
- `context.request(endpoint, options?)`

## 4. 推荐分层

远程核心脚本建议拆成：

1. `bootstrap(context)` 入口层
2. 页面适配层
3. 功能层
4. 远程策略读取层

建议把这些内容放到远程配置或远程模块中：

- 站点选择器
- 功能开关
- 文案和提示内容
- 规则参数
- 版本兼容策略

建议保留在本地页面执行层的内容：

- DOM 查询与写入
- 事件绑定
- 页面按钮交互
- 少量胶水逻辑

## 5. Manifest 建议结构

示例：

```json
{
  "site": "example.com",
  "selectors": {
    "primaryButton": ".submit-btn"
  },
  "featureFlags": {
    "showBanner": true
  }
}
```

运行时配置示例：

```json
{
  "bannerText": "Remote core module active",
  "selectors": {
    "primaryButton": ".submit-btn"
  },
  "featureFlags": {
    "showBanner": true
  }
}
```

## 6. 模块模板来源

后台“授权与运行时”页已提供远程模块模板填充能力。

模板骨架当前要求：

```js
export async function bootstrap(context) {}
export async function destroy(context) {}
export function getHealth() {}
```

## 7. 第一阶段到第二阶段的衔接

- 已有 `verified_loader` 脚本可继续沿用第一阶段模式
- 重点脚本可逐步切换到 `remote_core`
- 切换前需先保证：
  - 脚本已开启运行时增强
  - 发布模式已切到 `remote_core`
  - 已在后台维护 Manifest
  - 已上传并发布远程模块

## 8. 当前实现范围

当前版本已落地：

- 远程核心壳脚本下发
- Manifest 存储与发布骨架
- 远程模块存储、读取、发布骨架
- 模块模板返回
- Manifest 发布、激活模块切换、版本回滚、模块转草稿联动约束

后续继续完善：

- 远程配置灰度
- 多模块组合
- 远程模块签名校验
- 后台更完整的版本切换与回滚
