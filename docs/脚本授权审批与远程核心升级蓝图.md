# 脚本授权审批与远程核心升级蓝图

本文档是本项目后续实现“脚本授权审批、单机绑定、使用监控、远程核心脚本、多发布模式”的唯一执行蓝图。后续需求评估、数据库设计、接口实现、前后台改造、打包交付、部署验证，统一以本文为准；若有新增结论，优先更新本文，再继续开发。

当前进度说明：第一阶段“基础授权闭环”已完成首轮落地，现有代码已接入授权申请、审批、激活、单机绑定、会话心跳、`verified_loader` 增强下载和后台基础管理页；后续继续按本文推进第二阶段远程核心能力。

补充说明：第一阶段已额外落地 UTF-8 安全申请字段、授权停用/启用、设备重置等管理能力；脚本运行时提交中文申请信息时，应优先使用 Base64 安全字段，避免跨终端编码差异导致乱码。

当前进度补充：第二阶段已完成第一批骨架代码，包括 `remote_core` 壳脚本下发、Manifest 存储接口、远程模块存储/发布接口、模块模板返回接口，以及后台基础编辑入口；后续继续完善回滚、灰度、签名校验和更细化的版本管理。

进一步补充：当前版本已新增远程核心的发布状态联动能力，包括 Manifest 发布、Manifest 激活模块切换、按 Manifest 版本回滚，以及“已被当前发布 Manifest 使用的模块禁止直接转草稿”的约束，避免线上版本关系失控。

实战样例补充：已基于 `xuexi.user.js` 生成第一例 `remote_core` 改造样例，示例文件位于 `docs/examples/remote-core/xuexi/`，包含远程模块样例、Manifest 示例、运行时配置示例和导入说明；当前采用“最小入侵”策略，先整体包进 `bootstrap(context)`，后续再逐步继续抽离选择器、文案和策略参数。

样例深化补充：`xuexi` 首例样板已进一步把关键页面选择器、面板文案和学习参数抽离到 `runtimeConfig`，后续新脚本改造时可优先参考这一做法，先把“站点差异点”和“可调整参数”配置化，再逐步拆更细的业务模块。

## 1. 文档目标

### 1.1 本次升级要解决的问题

当前项目已经具备脚本、插件、EXE 小程序的分发能力，也支持脚本混淆、私密访问、同步、备份、在线更新，但对于脚本版权保护和合理使用监管仍有以下不足：

1. 单纯依赖前端混淆，仍然存在被逆向、二次传播、离线永久使用的风险。
2. 已分发脚本缺少统一授权审批机制，无法做到“申请 -> 审批 -> 绑定 -> 使用追踪”的完整闭环。
3. 脚本分发后缺少持续可控的运行反馈，后台无法稳定看到“谁在什么时间、在哪台电脑、用了多久”。
4. 未来服务器地址可能变更，脚本若直接写死服务端地址，后续维护成本高。
5. 新旧脚本缺少统一升级框架，无法对已上传脚本进行低成本、可批量的安全增强。

### 1.2 本次升级的最终目标

本项目升级后，脚本分发必须支持以下能力：

1. 每个脚本可独立选择发布模式：原始直装、混淆直装、验证增强、远程核心。
2. 每个脚本可独立选择授权策略：无需授权、申请审批后授权、绑定设备、限制有效期、强制停用。
3. 已上传脚本可在尽量不重写业务代码的前提下，接入统一运行时模块，实现傻瓜式增强。
4. 远程核心模式下，前台分发的只是壳脚本，关键规则、配置或核心逻辑由服务器控制。
5. 后台可查看授权申请、审批结果、设备绑定、在线状态、使用记录、异常记录。
6. 新功能必须在不破坏现有下载、同步、备份、更新、Docker/PM2 部署能力的前提下无损升级。

## 2. 升级原则

### 2.1 无损升级原则

1. 现有脚本、插件、小程序的上传、下载、同步、导出、在线更新能力必须保持可用。
2. 现有数据库必须通过增量迁移升级，不允许要求用户删除旧库重建。
3. 现有前台下载路径必须继续兼容旧链接，避免历史安装地址失效。
4. 新功能默认关闭，旧脚本不启用新策略时，应继续按旧模式运行。
5. 同步节点之间存在版本差异时，旧节点至少不能因新字段崩溃；同版本节点必须完整同步新功能数据。

### 2.2 多模式并存原则

每个脚本独立选择，不搞全站强制切换。至少保留以下四种模式：

1. `direct_raw`：原始脚本直接下载。
2. `direct_obfuscated`：混淆脚本直接下载。
3. `verified_loader`：本地脚本主体 + 统一运行时验证模块。
4. `remote_core`：壳脚本 + 服务端下发配置/模块/远程核心逻辑。

### 2.3 服务端优先控制原则

1. 关键授权决策必须由服务端返回，脚本本地只执行结果。
2. 授权是否有效、设备是否匹配、是否允许继续运行、是否要求升级，都由服务端控制。
3. 所有运行状态上报、审批结果、绑定关系、使用时长统计，以服务端记录为准。

### 2.4 运行时可迁移原则

1. 脚本中不得把单一服务器地址硬编码在多个业务位置。
2. 远程连接必须统一通过“运行时模块配置”读取。
3. 必须支持主地址 + 备用地址 + Manifest 中转地址三层配置。
4. 后续迁移服务器时，优先通过后台配置切换，不要求批量手改脚本正文。

## 3. 新功能范围

本次蓝图覆盖以下能力：

1. 脚本授权申请与审批。
2. 授权码管理。
3. 单设备绑定与设备换绑审批。
4. 脚本运行时统一模块。
5. 启动、心跳、结束、错误、拒绝等运行事件上报。
6. 脚本发布模式扩展。
7. 远程核心脚本架构。
8. 同步、备份、导出导入、在线更新的兼容扩展。
9. 新脚本开发规范与作者接入规则。

## 4. 总体架构

升级后脚本体系分为四层：

### 4.1 资源层

仍由现有 `scripts`、`script_versions`、上传文件、下载接口组成，负责保存脚本元数据和安装文件。

### 4.2 授权层

新增授权申请、授权码、设备绑定、审批记录等数据表与接口，负责“谁可以用、在哪台设备用、是否继续有效”。

### 4.3 运行时层

新增统一脚本运行时模块，负责：

1. 拉取全局运行时配置。
2. 请求授权校验。
3. 获取运行模式。
4. 获取远程配置或远程模块。
5. 上报启动、心跳、退出、错误、功能使用事件。

### 4.4 远程核心层

用于保存脚本远程配置、规则、模块、Manifest 和版本策略。远程核心模式下，本地脚本只保留最薄的一层壳逻辑。

## 5. 发布模式定义

### 5.1 `direct_raw`

用途：公开脚本、低敏感脚本、无需授权限制的脚本。

行为：

1. 下载原始脚本文件。
2. 不注入运行时模块。
3. 不要求审批和设备绑定。

### 5.2 `direct_obfuscated`

用途：继续兼容现有混淆下载模式。

行为：

1. 下载混淆后的脚本。
2. 可选要求访问密码或后台登录。
3. 不强制接入运行时授权。

说明：此模式保留，但不再作为高安全方案的主路线。

### 5.3 `verified_loader`

用途：用于已上传脚本的低成本安全升级。

行为：

1. 下载结果仍然是单个本地脚本文件。
2. 系统在脚本固定位置注入统一运行时模块。
3. 脚本启动前先执行授权校验、设备校验、运行上报。
4. 校验通过后再进入原脚本主体。
5. 可不改原业务逻辑，只在入口处进行包装。

说明：这是旧脚本批量增强的主方案。

### 5.4 `remote_core`

用途：用于重点脚本、高敏感脚本、核心逻辑需要服务端控制的脚本。

行为：

1. 前台安装的是壳脚本，不是完整业务脚本。
2. 壳脚本通过统一运行时模块获取远程配置与远程模块。
3. 关键规则、功能开关、版本策略、部分核心逻辑由服务端控制。
4. 授权无效、设备不匹配、版本被停用时，脚本不继续运行。

说明：此模式安全性最高，但要求脚本作者按新框架改造脚本。

## 6. 授权审批模型

### 6.1 业务目标

脚本不是自由下载后无限使用，而是按“申请审批、授权使用、单机绑定、持续反馈”的流程执行。

### 6.2 基础流程

1. 用户安装脚本。
2. 首次运行时，脚本发现未授权。
3. 脚本弹出申请入口，收集申请信息。
4. 后端生成授权申请记录。
5. 管理员在后台审批。
6. 审批通过后生成授权码，并设置授权字段。
7. 用户输入授权码，首次激活绑定设备。
8. 后续每次运行都进行授权校验与设备校验。

### 6.3 后台审批字段

授权审批至少支持以下字段：

1. 使用人姓名。
2. 联系方式。
3. 申请用途。
4. 申请备注。
5. 授权脚本。
6. 授权码。
7. 状态：待审批、已批准、已拒绝、已停用、已过期。
8. 允许设备数，默认 `1`。
9. 是否允许换绑。
10. 生效时间、过期时间。
11. 审批备注。
12. 审批人。

### 6.4 授权码命名要求

对外文案统一使用“授权码”，不得使用“卡密”文案。

## 7. 单设备绑定设计

### 7.1 目标

脚本一旦在一台电脑完成首次激活，就不能在其他电脑直接继续使用，除非后台审批换绑。

### 7.2 设备识别原则

浏览器脚本无法获取绝对硬件序列号，因此采用“设备指纹 + 本地持久标识”的组合方案：

1. 浏览器 UA。
2. 平台信息。
3. 时区、语言。
4. 屏幕分辨率。
5. Canvas/WebGL 等可用环境指纹。
6. 本地持久 UUID。
7. 站点环境特征。

服务端将其组合为 `device_fingerprint`，并存储摘要值而非过度暴露原始数据。

### 7.3 绑定策略

1. 首次激活成功后自动绑定当前设备。
2. 若授权允许设备数为 `1`，其他设备使用同授权码时必须拒绝。
3. 若检测到新设备，可生成“换绑申请”或“异常尝试记录”。
4. 后台支持手动解除旧绑定，再批准新设备绑定。

### 7.4 风险声明

此方案是“足够实用的合理限制”，不是绝对不可破解的硬件锁。产品定位必须明确为合理使用监控与权限控制，不宣传为绝对安全。

## 8. 使用监控与时长统计

### 8.1 必须采集的运行事件

1. `startup`：脚本启动。
2. `auth_success`：授权校验通过。
3. `auth_reject`：授权拒绝。
4. `heartbeat`：心跳。
5. `runtime_error`：脚本异常。
6. `feature_call`：关键功能调用，可选。
7. `shutdown`：正常结束，可选。

### 8.2 使用时长计算方式

1. 启动时创建会话。
2. 脚本定时上报心跳，建议默认 60 到 300 秒可配置。
3. 若超出指定超时时间未收到新心跳，则自动结束会话。
4. 使用时长 = 会话开始时间到最后一次有效心跳时间。

### 8.3 后台可视化要求

后台至少能看到：

1. 授权人最近活跃时间。
2. 当前在线设备数。
3. 每个脚本近 7 天、30 天启动次数。
4. 每次会话的大致使用时长。
5. 最近错误记录与授权拒绝原因。

## 9. 运行时模块设计

### 9.1 模块目标

所有需要授权、上报或远程核心能力的脚本，必须统一通过运行时模块连接服务器，而不是各写一套网络逻辑。

### 9.2 模块职责

运行时模块负责：

1. 读取全局运行时配置。
2. 计算设备标识。
3. 执行授权申请、激活、校验。
4. 获取远程 Manifest。
5. 获取远程配置与远程模块。
6. 创建会话并发送心跳。
7. 统一处理失败重试、备用地址、版本停用、升级要求。

### 9.3 固定注入位置

为了兼容旧脚本和便于后续升级，系统生成脚本时统一采用以下结构：

1. 第一段：UserScript 元信息头。
2. 第二段：运行时配置块。
3. 第三段：统一运行时模块加载代码。
4. 第四段：脚本主体入口或壳入口。

固定要求：

1. 运行时配置块必须放在元信息头之后、主体逻辑之前。
2. 旧脚本增强时，只允许在此固定区域注入，不在脚本多处散改。
3. 新脚本开发时必须预留 `bootstrap` 入口，由运行时模块调用。

### 9.4 运行时配置项

统一运行时配置至少包含：

1. `scriptId`
2. `version`
3. `releaseMode`
4. `runtimeProfile`
5. `manifestUrl`
6. `runtimeBaseUrl`
7. `runtimeFallbackUrls`
8. `heartbeatInterval`
9. `authRequired`
10. `allowOfflineGracePeriod`

### 9.5 地址迁移策略

后续迁服时按以下优先级解析：

1. 脚本内置 `manifestUrl`
2. Manifest 返回的当前主运行时地址
3. Manifest 返回的备用地址列表
4. 后台全局运行时地址配置

要求：

1. 远程核心脚本不依赖单一地址永久不变。
2. 若主地址失效，应自动回退到备用地址。
3. 更换服务器时，应优先通过后台配置和 Manifest 切换，不要求用户重装所有脚本。

## 10. 远程核心模式设计

### 10.1 适用场景

以下脚本适合远程核心模式：

1. 核心价值在规则、参数、配置、策略上的脚本。
2. 需要频繁调整策略、灰度开关、停用旧版本的脚本。
3. 需要强授权控制和持续在线验证的脚本。

### 10.2 不适合全部远程化的内容

以下内容最终仍要在浏览器执行：

1. DOM 查询与页面操作。
2. 事件监听。
3. 页面注入 UI。
4. 与网页上下文强耦合的前端逻辑。

因此远程核心的正确思路不是“所有代码都放服务器”，而是“把关键价值和可控逻辑尽量放服务器”。

### 10.3 远程核心组成

1. 壳脚本：本地安装文件。
2. Manifest：脚本当前可用配置与模块索引。
3. Remote Config：运行参数、功能开关、目标站点策略。
4. Remote Module：按版本发布的远程逻辑模块。
5. Auth Policy：授权与设备校验策略。

### 10.4 远程核心最小执行流程

1. 壳脚本启动。
2. 加载运行时模块。
3. 读取 Manifest。
4. 进行授权校验。
5. 获取 Remote Config。
6. 拉取 Remote Module。
7. 执行模块暴露的 `bootstrap(context)`。
8. 运行中定时心跳和状态上报。

## 11. 旧脚本升级策略

### 11.1 升级分级

旧脚本升级分为三档：

#### A. 兼容保留

保持现有直装或混淆直装逻辑，不接入新能力。

#### B. 一键增强

适用多数旧脚本：

1. 自动注入运行时模块。
2. 自动增加授权校验。
3. 自动增加运行上报。
4. 原脚本主体尽量不动。

这是“傻瓜式增强”的默认方案。

#### C. 深度改造

适用于重点脚本：

1. 按远程核心规范重构。
2. 把关键逻辑拆到服务端。
3. 本地脚本只保留壳和页面交互。

### 11.2 一键增强适配条件

满足以下条件的脚本可以优先做一键增强：

1. 有可读源码或未彻底丢失原始结构。
2. 主体逻辑可通过入口函数包裹。
3. 不依赖非常特殊的自执行时序。

### 11.3 不可自动化的场景

以下情况不得宣称可完全傻瓜式升级到远程核心：

1. 只有高度混淆产物，没有可维护源码。
2. 脚本主体大量依赖顶层立即执行且无明确入口。
3. 业务逻辑和 DOM 逻辑完全混在一起，无法抽离。

## 12. 新脚本作者接入规范

本节是后续脚本作者必须遵守的开发规则。

### 12.1 统一入口规则

新脚本必须采用统一入口结构，不再允许整份脚本直接在顶层散乱执行。标准结构为：

1. 元信息头。
2. 运行时配置块。
3. `main(context)` 或 `bootstrap(context)` 入口。
4. 业务模块按函数或对象拆分。

要求：

1. 所有业务逻辑必须从统一入口启动。
2. 不得把网络地址写死在多个业务函数中。
3. 不得在脚本多处直接拼接授权接口地址。
4. 所有远程访问必须走运行时提供的 API。

### 12.2 运行时上下文要求

运行时传给脚本作者的 `context` 至少包括：

1. 当前脚本元信息。
2. 用户授权信息摘要。
3. 当前设备信息摘要。
4. 远程配置。
5. 统一日志接口。
6. 统一请求接口。
7. 生命周期钩子注册能力。

### 12.3 新脚本开发分层要求

新脚本必须按下列分层组织：

1. `runtime shell`：运行时桥接层。
2. `site adapter`：站点选择器和页面适配层。
3. `feature layer`：业务功能层。
4. `remote policy`：由服务端返回的规则与策略层。

### 12.4 推荐的作者改造规则

脚本作者在适配远程核心模式时，应优先把以下内容抽到服务端：

1. 规则清单。
2. 接口参数映射。
3. 功能开关。
4. 版本兼容策略。
5. 高价值判断逻辑。
6. 可频繁变更的策略数据。

保留在本地的内容主要是：

1. 页面读写。
2. UI 交互。
3. 事件监听。
4. 少量必要胶水逻辑。

### 12.5 新脚本远程核心接口约定

远程模块至少导出：

1. `bootstrap(context)`
2. `destroy?(context)`
3. `getHealth?(context)`

要求：

1. `bootstrap` 必须幂等。
2. 远程模块加载失败时，必须允许运行时安全退出，不得无限报错刷屏。
3. 远程模块版本必须与 Manifest 中声明版本一致。

## 14. 后端接口蓝图

### 14.1 后台管理接口

建议新增路由模块：`server/src/routes/runtime.js` 或 `server/src/routes/license.js`，统一提供以下接口：

1. 运行时全局配置管理。
2. 脚本发布模式配置管理。
3. 授权申请列表、审批、拒绝。
4. 授权码列表、新建、停用、延期、删除。
5. 设备绑定列表、解绑、换绑审批。
6. 使用会话、事件日志查询。
7. Manifest 和远程模块管理。

### 14.2 脚本运行时接口

建议至少提供以下 API：

1. `POST /api/runtime/auth-request`
   - 提交授权申请。
2. `POST /api/runtime/activate`
   - 使用授权码激活并绑定设备。
3. `POST /api/runtime/validate`
   - 运行前校验授权、设备、版本、模式。
4. `POST /api/runtime/session/start`
   - 创建会话。
5. `POST /api/runtime/session/heartbeat`
   - 上报心跳。
6. `POST /api/runtime/session/end`
   - 正常结束会话，可选。
7. `POST /api/runtime/event`
   - 上报错误、功能调用、拒绝原因等。
8. `GET /api/runtime/manifest/:scriptId`
   - 获取远程 Manifest。
9. `GET /api/runtime/config/:scriptId`
   - 获取远程配置。
10. `GET /api/runtime/module/:scriptId/:version`
   - 获取远程模块。

### 14.3 下载接口兼容要求

现有 `server/src/routes/download.js` 必须改造成支持多模式分发：

1. 旧模式继续返回原始或混淆脚本。
2. 验证增强模式返回“运行时注入版脚本”。
3. 远程核心模式返回“壳脚本”。
4. 老的安装链接和下载统计逻辑必须继续可用。

## 16. 同步与多节点适配要求

### 16.1 必须同步的新数据

若节点开启同步，至少要支持以下新增数据同步：

1. 运行时配置。
2. 脚本发布模式元数据。
3. 授权申请记录，可按策略决定是否同步。
4. 授权主体与设备绑定。
5. 远程 Manifest。
6. 远程模块元数据和文件。

### 16.2 同步策略建议

按敏感度分层：

1. 公共脚本元数据、Manifest、远程模块版本信息：可同步。
2. 授权申请、授权码、设备绑定、运行事件：默认不跨站公开同步，需显式配置。
3. 若是主从站模式，授权与运行数据应优先归集到主站，由主站下发最终授权决策。

### 16.3 拉取推送兼容原则

现有 `sync` 路由必须升级但不破坏旧能力：

1. 新字段不影响原有分类、脚本、插件、程序同步。
2. 若对端节点不支持新表，新版节点不能因此崩溃。
3. 同版本新节点之间，必须可同步运行时配置、远程核心模块和相关文件。
4. 文件同步路径必须继续兼容 Docker 与 PM2 两套环境。

### 16.4 文件存储建议

新增远程核心文件时，建议在 `uploads` 下增加受控子目录，例如：

1. `uploads/runtime-manifests/`
2. `uploads/runtime-modules/`

要求：

1. 必须继续走现有安全路径校验思路。
2. 下载回源与同步补拉必须适配这些新目录。

## 17. 备份、导入导出与在线更新适配

### 17.1 备份导出要求

`backup` 功能后续必须包含：

1. 新增数据表数据。
2. 运行时相关文件。
3. 远程 Manifest 与远程模块文件。
4. 授权、审批、设备绑定、会话、事件等记录。

### 17.2 导入恢复要求

1. 旧版本备份导入到新版本时，缺失新表数据视为允许，但不能崩溃。
2. 新版本备份导回同版本时，需完整恢复新功能数据。

### 17.3 在线更新要求

本项目后续所有相关开发，必须支持通过现有在线更新能力无损升级，要求如下：

1. 修改只落在允许目录：`server/src/`、`client/src/`、`docs/`、必要时 `scripts/`。
2. 打包必须继续只走 `node scripts/create-update.js`。
3. 更新包必须继续包含 `client/dist`，不得破坏现有更新流程。
4. Docker 环境必须继续把后端更新到 `/app/src`，前端产物更新到 `/app/public`。
5. PM2/非 Docker 环境必须继续把前端产物更新到 `client/dist`。
6. 新增运行时文件与远程核心目录时，要确保在线更新后旧数据目录不被误删。

### 17.4 更新后验证清单

每次交付必须至少验证：

1. 首页和后台能打开。
2. 脚本、插件、程序下载仍正常。
3. 原始、混淆、验证增强、远程核心四种模式至少抽样验证。
4. 授权申请、审批、激活、单机绑定、心跳上报正常。
5. 同步节点基础拉取推送正常。
6. Docker/PM2 中的在线更新、回滚、导出导入仍正常。

## 18. 分阶段实施计划

### 第一阶段：基础授权闭环

目标：先实现最小可用版本。

范围：

1. 数据库新增授权申请、授权码、设备绑定、会话、事件表。
2. 脚本管理页增加发布模式和授权开关。
3. 新增运行时基础模块。

5. 后台可审批授权申请，可查看基本运行记录。

交付标准：

1. 一个旧脚本可在不重写主体的前提下接入授权与监控。
2. 能做到申请、审批、授权码激活、单机绑定、启动和心跳上报。

### 第二阶段：远程核心能力

范围：

1. 增加 Manifest、远程配置、远程模块管理。

3. 增加全局运行时地址和备用地址机制。
4. 增加旧版本停用、强制升级能力。

交付标准：

1. 至少一个重点脚本完成远程核心化改造。
2. 迁服时只需改运行时配置，不需重做全部脚本。

### 第三阶段：同步、报表、精细化管理

范围：

1. 同步支持运行时新数据。
2. 完善使用报表和告警视图。
