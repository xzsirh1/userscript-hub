# 远程核心脚本书写规范

## 目标

让脚本尽量可以被后台自动包装为远程核心模块，同时尽量保留原有行为。

## 推荐写法

- 保留完整的 `// ==UserScript==` 头部。
- 把稳定的选择器、文案、节奏参数放进可配置项。
- 主要逻辑保持在单一脚本主体中，先跑通，再逐步抽离。
- 需要远程核心时，优先导出 `bootstrap(context)`，可选 `destroy(context)` 和 `getHealth()`

## 适合自动梳理的内容

- 整段脚本主体自动包进模块草稿。
- 头部信息自动保留。
- 常见 `GM_*`、网络请求、定时器、选择器、文案会被自动识别并给出提示。

## 需要人工复核的内容

- `eval`、`new Function`、`document.write`
- `unsafeWindow`
- 模块语法混入普通用户脚本主体
- 依赖页面结构但没有抽成配置的选择器

## 兼容性边界

- 不承诺任何脚本都能 100% 自动无损拆分。
- 自动化默认采用“最小入侵”策略，先保证能跑，再提示需要整理的部分。
- 如果脚本本身已经很复杂，应该先按运行配置分层，再考虑更细的模块化。

## 给 AI 的写法建议

- 不要擅自改掉原头部语义。
- 不要删除原脚本已有的授权、匹配、资源声明。
- 先保留功能，再把可变项外置到运行参数中。
- 若发现高风险写法，优先输出提醒而不是强行改写。
